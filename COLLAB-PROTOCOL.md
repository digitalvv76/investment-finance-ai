## 0. 角色
- **V1 窗口** = `v1-stable` 分支 @ `D:\class1\.claude\worktrees\v1-stable`。定位：紧急热修、推送效果观察、调参。

## 1. 单一部署路径（消除撞车）⚠️V2待复核
- 生产**只走 git**：`git push origin main` → 服务器 `git pull` → rebuild。
- **V1 窗口不再 scp/deploy.sh 覆盖生产主机** —— 那会被 main 的 git 操作冲掉（今天 11:30 的事故）。
- V1 需类生产验证 → 用**影子/金丝雀容器**，不碰 live。

## 2. main 是主干，V1 不长期分叉 ⚠️V2待复核
- 新功能一律落 main。
- `v1-stable` **只用于"必须先上、等不及 main"的紧急热修**；修完**立刻 port 回 main，不积累**（今天分叉到 97/82 commit = 重复劳动之源）。

## 3. 交接/port 前先验证目标架构（V1 自律，即时生效）
- 任何 port 或写交接单前，**先 `git show <目标分支>:<文件>` 看目标真实代码**，别假设。
- 今天教训：基于 v1 内联架构写 HANDOFF 假设能 cherry-pick，没发现 main 已是流水线 → 差点塞死代码。V2 catch 了。

## 4. 开工前认领，避免重复 ⚠️V2待复核
- 会话开始先读对方 `SESSION.md` + `git log origin/main` 和 `origin/v1-stable`，确认没重叠再动手。
- 今天两窗口各自建了看门狗/时区修复/zhibo = 白干。

## 5. 按归属分工，尊重 owner（V1 自律）
- **谁拥有某子系统，谁是其内部实现的权威**；另一方只给**意图/契约**，不插手实现。
- 今天成功范例：V2 拥有 main 流水线 → 写接线层；V1 设计了安全网 → 写语义层(`SPEC-safety-net-pipeline.md`)。
- **不闯对方 worktree**：跨分支改动只经"交接文档 + 归属窗口"。V1 拒绝在本窗口 cherry-pick 进 main = 正确。

## 6. 分工搞混时：先提醒，不直接执行（V1 自律，用户明确要求）
- 当前窗口收到**可能属于另一个窗口/另一分支**的任务 → **不立即执行**，先提醒用户：说明"为什么像是另一窗口的活" + 给选项(①我出交接文档 ②你去那个窗口跑 ③你确认后我在这里做)。
- 触发情形：改/提交/cherry-pick 到 main 或 `D:\class1`；部署生产；动 main 流水线接线；建可能重复的大功能。
- **清楚是本窗口的活 → 照常直接做，不啰嗦。**

## 7. 文件式异步交接（继续保持）
- 一个主题一份文件，提交进仓库当信道（比人肉转述可靠、可审、持久）。
- 命名约定：`HANDOFF-*`（交接任务）、`ARCH-*`（架构事实）、`SPEC-*`（语义契约）、`DESIGN-*`（设计笔记）。

## 8. 最高杠杆（长期建议）⚠️需用户 + V2 拍板
- 双线本身是大多数摩擦的根源。**建议收敛成"单主干 main + 影子金丝雀验证"，退役 v1-stable 常驻分支**，用金丝雀做安全灰度而非永久并行分支。生产已成 main，此为自然终点。
- **[已决定 2026-07-10]** 用户拍板收敛退役。V2 评估：main 已含 v1-stable 全部有价值改动，无需再 port（详见 `RETIRE-v1-stable.md`）。分支保留归档、不再开发；worktree 待 V1 收工后移除。
