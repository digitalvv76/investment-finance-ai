# 工作流评估

> 2026-07-17。V1 窗口，对当前 8 条工作流的逐项评估。

---

## 评估框架

每条工作流按三个维度打分（🟢 健康 / 🟡 有摩擦 / 🔴 阻塞），并标注根因和改进方向。

---

## 1. 会话生命周期

**评分**：🟢 健康（7/10）

| 环节 | 状态 | 说明 |
|------|------|------|
| SessionStart hook 链 | 🟢 | startup.py → backup → verify_env → first response rule，四层覆盖完整 |
| 会话中 HISTORY.md 更新 | 🟢 | 实时追加，commit hash 引用到位 |
| SessionEnd 检查清单 | 🟡 | dev_checklist.py 跑**全量** 525 tests（~200s），而 pre-commit 已证明**定向测试**更高效。全量跑只在 session end 有意义，但慢 |
| ChromaDB Windows 容错 | 🟡 | 6-7 个 test_vector_store 失败被容忍，靠 `-rE` 逐行验证。当前有效，但脆弱——如果某天真出现非 ChromaDB 的 Windows 错误，会被误吞 |
| HISTORY.md 滞后 | 🟡 | session_end_backfill.py 按 hash 机械补录 commit，但**非 commit 决策**（跳过/确认/方向调整）不自动补录。依赖 AI 手动写 |

**改进方向**：
- SessionEnd 也改用定向测试（只跑本次会话修改的模块），全量测试留给 CI/定期
- 给非 commit 决策一个快速记录机制（一句话 + 标签），session end 自动聚合

## 2. 开发流程（idea → 部署）

**评分**：🟢 健康（8/10）

| 环节 | 状态 | 说明 |
|------|------|------|
| Spec → 实现 → 测试 → 部署 | 🟢 | 链路清晰，spec 命名规范（日期-序号-描述）|
| 模块注册（__manifest__.json）| 🟢 | 强制注册，pre-commit 拦截未注册模块，杜绝孤儿代码 |
| Pre-commit 定向测试 | 🟢 | 只跑被修改模块的测试，比全量快 5-10x |
| `__ALL__` 触发全量 | 🟡 | manifest 里 `tests: ["__ALL__"]` 会把定向测试变成全量 525 tests。不确定哪些模块设了 `__ALL__`，触发频率未知 |
| 质量闸门触发 | 🟡 | 全靠 AI 自行判断"是否高风险"。没有自动化触发——同一个改动，不同 AI 可能不同判断 |
| 代码审查 | 🟢 | 对抗式核实已嵌入流程，同模型盲点认知到位 |

**改进方向**：
- 审计一次所有 `__ALL__` 标记，确认每个都有充分理由；没有理由的缩到具体 test 文件
- 给高风险判断加一个简单 checklist（碰了 dispatch.py？碰了 schema？碰了 deploy？），去掉 AI 主观判断

## 3. V1/V2 协作

**评分**：🟡 有摩擦（6/10）

| 环节 | 状态 | 说明 |
|------|------|------|
| 分工边界 | 🟢 | COLLAB-PROTOCOL.md 定义清晰，V1 决策+需求，V2 代码+部署 |
| 异步交接 | 🟡 | 文件式交接（HANDOFF-* / SPEC-*）靠人工发现。V1 写完 spec，V2 **不知道**——除非 V2 主动 `git pull` 并 grep 新文件。没有通知机制 |
| v1-stable 分支 | 🔴 | 设计为"短期热修复通道"，实际已存活数周。过去积累过 97-commit 分叉。COLLAB-PROTOCOL 虽规定"短命+立刻回迁 main"，但**缺乏强制执行**——靠人遵守 |
| 重复劳动 | 🟡 | 历史上发生过（双方独立建 watchdog/修时区），协议已修补但无法根除——两个窗口各自独立工作，信息不对称是结构性的 |
| 跨窗口提醒 | 🟢 | memory [[v1-v2-confirm-before-crossing]] 生效，跨窗口任务先提醒确认 |

**改进方向**：
- **交接通知**：V1 写完 spec 后，在 SESSION.md 顶部加一个 `## 🆕 待 V2 读取` 段，V2 启动时 session_startup.py 自动展示
- **v1-stable 寿命限制**：每次 V1 会话启动时，检查 v1-stable 是否落后 main 超过 N 个 commit，超过就强制提醒合并
- **或者干脆取消 v1-stable**：V1 直接在 main 上写 spec 文档（不写代码），消除分叉压力。这与"V1 不能改代码"一致——spec 是文档不是代码

## 4. Git 工作流

**评分**：🟢 健康（7/10）

| 环节 | 状态 | 说明 |
|------|------|------|
| Commit 规范 | 🟢 | `type(scope): description` 格式统一，`[skip-tests]` 紧急通道可用 |
| Pre-commit 门禁 | 🟢 | 格式检查 + manifest 门禁 + 定向测试，三层到位 |
| Pre-push 保护 | 🟢 | `pre_push_check.py` 阻止直接推 v1-stable |
| 分支策略 | 🟡 | main（主干）+ v1-stable（热修复）+ feature/fix 分支。实际 v1-stable 不是"短期"的 |
| 合并 main→v1-stable | 🟡 | 手动操作，容易忘记。上次同步 60 个落后提交 |

**改进方向**：
- 见 V1/V2 协作——v1-stable 的根本定位需要重新决策

## 5. 部署流程

**评分**：🟢 健康（8/10）

| 环节 | 状态 | 说明 |
|------|------|------|
| deploy-main.sh | 🟢 | 5 步标准流程：UptimeRobot 暂停 → 回滚 tag → git sync → 重建 → 健康验证。`trap EXIT` 保证监控恢复 |
| 回滚机制 | 🟢 | 每次部署自动 `docker tag` 留快照，回滚一条命令 |
| 影子部署 | 🟢 | deploy-shadow.sh，独立容器+独立数据卷+DRY_RUN_PUSH，隔离保证到位 |
| SRC_PATHS 硬编码 | 🟡 | deploy-main.sh 里源文件列表是手写的。新增模块如果不在列表里，部署时不会同步到 ECS。当前靠人记住 |
| 部署哨兵 | 🟡 | Spec 写了但评估不通过（0/5 检查命中真实故障），决定不建。判断合理但文档分散——spec 在一个文件，决策在另一个 |
| 无灰度发布 | 🟡 | 全量部署，没有 canary/逐步放量。对于单容器单用户的系统，灰度可能过度设计——但值得记录这个取舍 |

**改进方向**：
- SRC_PATHS 改从 `__manifest__.json` 自动生成，消除手动维护
- 部署哨兵的决策记录合并到 deploy-main.sh 注释里，避免"为什么没做"的疑问

## 6. 质量保证

**评分**：🟢 健康（7/10）

| 环节 | 状态 | 说明 |
|------|------|------|
| 定向测试（pre-commit）| 🟢 | 快、准、不碍事 |
| 全量测试（session end）| 🟡 | 慢（~200s），与定向测试重复。见会话生命周期 |
| 看门狗 | 🟢 | 独立 asyncio task，不会被调度器 hang 死。四态判断（STALLED/DEGRADED/QUIET_OK/HEALTHY），告警去抖+冷却到位 |
| 对抗式核实 | 🟡 | 效果好但**全手动触发**。没有"高风险改动 → 自动建议核实"的提示机制 |
| 回滚 tag | 🟢 | 每次部署自动创建 |
| 部署后验证 | 🟡 | 依赖健康检查 + 看门狗 + UptimeRobot + 用户观察。部署后没有**主动**跑一个烟雾测试（发一条测试推送/查一条最新数据） |

**改进方向**：
- 部署后自动查"最近 5 分钟是否有新采集数据入库"——比健康检查 endpoint 更接近真实业务健康
- 对抗式核实的触发条件可以从"AI 自行判断"改为"改了这些文件就建议核实"（维护一个高风险文件清单）

## 7. 任务追踪

**评分**：🟡 有摩擦（5/10）

| 环节 | 状态 | 说明 |
|------|------|------|
| HISTORY.md | 🟢 | 跨会话真相，追加不覆盖，commit hash 引用到位 |
| SESSION.md | 🟡 | "当前状态"靠一段自由文本。任务没有 ID、没有状态机（pending→in_progress→done）、没有阻塞关系。跨会话时，**上个小会话的"下一步"可能被遗忘**——除非 AI 主动搬运 |
| 待办追踪 | 🔴 | V1 有多个"待 V2 审查/实施"的事项（资金流信号、训练体系、富途 P3），散落在 HISTORY.md/SESSION.md/口头讨论中。**没有一个地方能看到所有 open tasks** |
| TROUBLESHOOTING.md | 🟢 | ~22 条目，症状→根因→修复→教训，格式统一 |
| Memory 文件 | 🟢 | 结构化，按主题分文件，wikilink 互联 |

**核心问题**：任务追踪靠"人（AI）记住并手动搬运到下一会话的 SESSION.md"。这在会话量大、任务多时不可靠。

**改进方向**：
- 引入 `TASKS.md`（借鉴 OpenAlice 的 Issue 理念）：每个任务一行，状态+优先级+来源会话
- 或者沿用上期讨论的 `.claude/issues/` 目录方案
- 最低成本方案：SESSION.md 里的"下一步"改成表格格式（至少保证结构化的待办不被遗漏）

## 8. 事故响应

**评分**：🟢 健康（7/10）

| 环节 | 状态 | 说明 |
|------|------|------|
| 检测层 | 🟢 | 看门狗（内部）+ UptimeRobot（外部）+ 用户观察，三层覆盖 |
| TROUBLESHOOTING.md | 🟢 | 22 条记录，格式规范，每次事故后立即追加 |
| 诊断流程 | 🟡 | V1 诊断 → 写 spec → V2 修复 → 部署。V1 能读代码但不能改——诊断和修复之间有**交接延迟**。紧急情况下（如采集停摆），这个延迟可能是致命的 |
| 回滚 | 🟢 | 一条命令，秒级恢复 |
| 事故复盘 | 🟡 | TROUBLESHOOTING.md 记录了"什么/为什么/怎么修"，但没有记录"**下次怎么更快发现**"和"**是否可能再次发生**" |

**改进方向**：
- TROUBLESHOOTING.md 每条增加两个字段：**检测改进**（下次怎么更快发现）和**复发可能性**（同样的事还会发生吗）
- 紧急情况给 V1 一个"直接 restart 容器"的权限——不算代码改动，但能救命（当前 V1 "不能碰 ECS"的铁律在紧急时可能是障碍）

## 汇总

| # | 工作流 | 评分 | 核心问题 |
|---|--------|------|---------|
| 1 | 会话生命周期 | 🟢 7/10 | SessionEnd 全量测试慢；非 commit 决策不自动补录 |
| 2 | 开发流程 | 🟢 8/10 | `__ALL__` 过度触发；质量闸门全靠 AI 判断 |
| 3 | V1/V2 协作 | 🟡 6/10 | 异步交接无通知；v1-stable 寿命失控 |
| 4 | Git 工作流 | 🟢 7/10 | v1-stable 定位需重新决策 |
| 5 | 部署流程 | 🟢 8/10 | SRC_PATHS 手动维护；无灰度（合理取舍）|
| 6 | 质量保证 | 🟢 7/10 | 对抗式核实手动触发；部署后无主动烟雾测试 |
| 7 | 任务追踪 | 🟡 5/10 | **无结构化任务管理**，待办散落各处 |
| 8 | 事故响应 | 🟢 7/10 | V1 诊断→V2 修复有延迟；复盘缺"检测改进"维度 |

**整体**：工作流基础扎实——会话持久化、部署自动化、质量闸门、事故响应都有成熟机制。摩擦集中在两个结构性问题上：

1. **V1/V2 双窗口的信息不对称**（协作延迟、任务遗忘、分支分叉）
2. **任务追踪缺乏结构化**（待办散落、跨会话丢失）

这两个问题相互放大：任务追踪越乱，双窗口越容易重复劳动或漏任务。

## 优先级建议

| 优先级 | 改什么 | 成本 | 收益 |
|--------|--------|------|------|
| P0 | **TASKS.md 或结构化待办** | 1h | 根治任务散落，最低成本最大收益 |
| P0 | **V1 spec 写完 → SESSION.md 顶部标记** | 30min | V2 启动自动看到新交接，消除"不知道有活" |
| P1 | 审计 `__ALL__` 标记，缩到具体文件 | 1h | pre-commit 测试更快 |
| P1 | SRC_PATHS 从 manifest 自动生成 | 2h | 消除手动维护，防部署遗漏 |
| P2 | v1-stable 去留决策 | 讨论 | 结构性解决分叉问题 |
| P2 | 事故复盘增加"检测改进"字段 | 30min | 长期积累更有价值 |
