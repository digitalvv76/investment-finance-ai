# 设计规格：多 agent 开发审查流水线（dev-review pipeline）

> 日期：2026-07-10 ｜ 起草：V1 窗口（头脑风暴产出）｜ **实现归属：V2 窗口 / main 主干**
> 分工：V1 出设计/语义/契约（本文），V2 出实现（agent 定义 + 命令 + 编排）。按 `COLLAB-PROTOCOL.md`。

## Context（为什么做）
用户是金融专家、不碰技术、把开发全权委托给 AI。当前模式"AI 开发、用户做业务决策"缺一道**自动质量把关**——用户看不出技术隐患，只能靠口头汇报判断对错。今天(2026-07-10)已多次暴露：架构盲点、时区 bug、安全隐患需要人来 catch。
目标：给开发结果加一条**"审查三角"流水线**（安全 + 正确性 + 产品验收），自动把关、自动修复，只在真正需要业务决策时才打扰用户。**框架优先，先搭起来再完善。**

## 已锁定的决策（头脑风暴结论）
1. **目标**：质量/安全 + 减少人工验证 + 吞吐，三者都要，**先搭框架**。
2. **触发**：**会话内按需编排**（我完成一块开发就自动派审）。先轻后重——首期不做 CI/hook 常驻门禁。
3. **发现问题后**：**自动修复回环**——审出问题→主 agent 自动修→重审，直到通过；只有业务取舍/无法自动解决才升级给用户。
4. **正确性审查是标准配置**（非可选）。

## 角色（4 个 agent）
| 角色 | 职责 | 复用 |
|------|------|------|
| **主开发 agent**（主循环） | 设计 + 开发 + 编排回环 + 汇总 | —— |
| **安全审计 agent** | 凭证泄露、注入、危险/越权操作、依赖风险 | 内置 `/security-review` 技能 |
| **正确性审查 agent** | bug、逻辑错误、边界、回归风险 | 内置 `/code-review` 技能 |
| **产品验收 agent** | 结果是否满足需求/意图（功能完整、无跑偏） | **新建**（无现成，见下） |

## 流水线流程（会话内）
```
主 agent 开发完一块(有 diff)
  → 并行派出：安全审计 ∥ 正确性审查 ∥ 产品验收
  → 汇总三方 findings（去重、按严重度排序）
  → 主 agent 自动修复 → 重派受影响的审查 …循环
  → 三道全过 → 交付用户 + 一页小结(审了什么/修了什么/结论)
  → 升级给用户 仅当：业务取舍 / 需求二义 / 回环 N 次仍不过 / 审查方要求人判断
```

## 产品验收 agent 的"需求"从哪来（关键契约）
- **有 spec 的功能** → 核对 `docs/superpowers/specs/` 里对应设计文档（头脑风暴流程本就产出，天然当需求文档）。
- **临时任务** → 核对任务陈述的意图 + 用户已知业务目标（推送质量、关注股覆盖、手机严格性等）。
- 产出：`满足 / 部分满足(缺什么) / 跑偏(哪偏了)` + 具体缺口清单。
- 约定：**以后重要功能先走头脑风暴出 spec**，产品 agent 才有据可依。

## 回环与升级语义（防跑飞）
- 自动修复回环**上限 N=3**；超限→停，把剩余问题升级给用户。
- **升级触发**：业务取舍 / 需求二义 / N 次不过 / 任一审查方标注"需人判断" / 安全审计出**高危**（高危一律先报人，不自动改）。
- 每轮只重派**受本轮修复影响**的审查，省 token。

## 成本与适用范围（诚实约束）
- 每个子 agent = 独立一次分析，**多花 token**。
- **适用**：改推送逻辑、动数据库/schema、上线前、安全敏感、跨模块改动。
- **不适用（跳过全套）**：改文案、typo、纯文档、单行修复——主 agent 自行判断，可只跑其中一两道或全跳。

## 首期实现范围（轻版，V2 在 main 落地）
1. **产品验收 agent**：新建 agent 定义（`.claude/agents/product-evaluator.md`）或等价 prompt——输入 diff + 对应 spec/意图，输出 满足度 + 缺口。
2. **安全/正确性**：直接复用内置 `/security-review`、`/code-review`，不重造。
3. **`/dev-review` 命令**（`.claude/skills/dev-review.md`）：一键对当前 diff 跑三道审 + 汇总；回环由主 agent 编排。
4. 首期**手动编排**（主 agent 调度），不写确定性脚本。

## 升级路径（未来，不在首期）
- **重版**：用 Workflow 编排工具把"开发→并行审→循环到干净→汇总"写成**确定性脚本**（可并行、可循环、可复现）。
- **门禁版**：接 git hook / CI，提交/上线前自动触发（需常驻基础设施）。
- 触发升级的信号：轻版被证明有价值且高频使用。
