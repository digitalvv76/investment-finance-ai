# 设计规格：轻量开发质量把关（原"多 agent 流水线"瘦身版）

> 日期：2026-07-10 ｜ 起草：V1 窗口 ｜ 版本：**v2（经独立第三方评审后瘦身，替换 v1 的首期范围）**
> v1 原方案（4-agent 全套流水线）见文末「未来升级路径」——**不是首期范围**。

## Context（为什么做）
用户是金融专家、不碰技术、把开发全权委托给 AI。缺一道**自动质量把关**——用户看不出技术隐患，只能靠口头汇报判断对错。今天(2026-07-10)已暴露架构盲点、时区 bug、安全隐患需人来 catch。

## 独立评审结论（为什么从"4-agent 全套"瘦身）
以独立开发专家视角复评 v1 方案，四点促成瘦身：
1. **同模型 agent 不真正独立**：4 个 agent 底层同一模型，共享盲点。今天 `disable/silent` 语义错 **V1+V2 双双栽**就是铁证。真杠杆是**上下文多样性 + 对抗性证伪**，不是"贴角色名"。
2. **产品验收 agent 最悬**：依赖清晰实时 spec；本项目需求多为隐性、对话中演进 → 易变橡皮章或幻觉缺口。
3. **单人项目 ROI 不划算**：4-agent 门禁让每次审查成本翻几倍；而今天的 bug 都靠**观察+看门狗**几分钟内发现、靠**回滚 tag** 秒回。低成本兜底已覆盖大部分实际风险。
4. **回环可能自欺**：同模型修自己实例挑的问题，可能收敛到"看着对其实错"。**测试/可观测行为**是比"agent 共识"更硬的地面真值。
> 附加：今天真正的痛多是**流程/协调**（部署撞车、双线漂移），审查 agent 拦不住——那归 `COLLAB-PROTOCOL`。别用多 agent 对错问题下药。

## 首期范围（瘦身版，三件事）
### 1. 风险闸门：只在高风险改动上启动
**触发**（满足任一）：碰**推送逻辑 / 数据库·schema / 部署·上线 / 安全敏感(凭证·认证) / 跨模块**改动。
**跳过**：文案、typo、纯文档、单行/局部无风险修复。主 agent 自行判定，宁可漏跑一道也不给琐碎改动上仪式。

### 2. 一道"对抗式核实"（核心，取代 4 个角色 agent）
不是按角色分 4 个 agent，而是**派 1 次独立子 agent 做对抗式验证**，关键在**框架而非命名**：
- 子 agent 被明确要求：**默认怀疑、去证伪这个改动**；**拿语义对着代码核实**（如 `disable/silent/skip`/时区/边界到底是什么，别望文生义）；**找出会让它出错的场景**。
- 借现成 `/code-review` + `/security-review` 作两个视角（正确性 + 安全），但以**对抗框架**跑，不走过场。
- 输出：确认的问题清单（去噪、按严重度）。**高危/安全项先报人，不自动改。**

### 3. 主安全押在更硬更便宜的东西上（强化既有，非新造）
- **测试**：高风险改动**必须有覆盖该行为的测试**（今天时区 bug 正是 TDD 回归锁住的）。这是首要地面真值。
- **看门狗**：已上线，监控上游存活——留着。
- **回滚 tag**：部署前 `docker tag ... :rollback-<标签>`（V2 已在用）——**固化为部署标准动作**。
- **观察窗口**：部署后现场查日志确认（今天多次这么做）。

### 自动修复回环（收窄）
只对**明确的、有测试佐证的**问题自动修 + 重验；模糊的/需判断的**直接报人**。**上限 2 轮**，超限升级。

## 明确不做（首期）
- ❌ 4 个正式角色 agent（安全/正确性/产品/开发分立）
- ❌ 产品验收 agent —— 等**真的开始严格维护实时 spec** 再说
- ❌ `/dev-review` 命令 + 决策面板 + Workflow 确定性脚本
- ❌ 常驻 / CI 门禁

## 归属（瘦身后大幅缩小）
瘦身版**几乎不需要新建框架代码** —— 主体是**主 agent 的工作纪律 + 复用现成技能 + 强化测试/回滚习惯**，**任一窗口立即可执行、无需 V2 建东西**。唯一需在 main 固化的：把**回滚 tag** 写进部署脚本标准动作（V2 已在做，收尾即可）。

## 未来升级路径（用证据换，非首期）
只有当出现「**测试+看门狗+回滚兜不住的真实事故**」或「高风险改动频繁到手动编排累了」，才逐步加码：
- **v1 全套**：4 角色 agent（+产品验收，前提是 spec 纪律已建立）
- **确定性编排**：Workflow 脚本（并行审 + 循环到干净）
- **常驻门禁**：git hook / CI
每级升级都要有"上一级不够用"的**具体证据**，不预先建。

## 验收（框架自身）
拿今天真实案例回放：把 `disable/silent` 语义错的那段改动喂进"对抗式核实"，看它是否会**要求对着代码确认 disable 到底是静音发还是跳过** → 能 catch 即达标。（这正是 4-agent 全套里同模型双双漏掉、而"对着代码证伪"能拦下的那类错。）
